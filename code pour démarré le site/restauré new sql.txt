Pour remettre ton backup sur une nouvelle base PostgreSQL (après recréation), ça dépend surtout du format de ton fichier (souvent .sql ou .dump/.backup ). Voilà la procédure complète (Windows/PowerShell).

## 1) Recrée ton PostgreSQL et récupère la connexion
Quand tu recrées la base (Render/Supabase/Railway/Neon/ElephantSQL/etc.), récupère :

- Host
- Port (souvent 5432)
- Database name
- User
- Password
- (SSL requis ou non)
Souvent on te donne directement une URL :

- postgresql://USER:PASSWORD@HOST:5432/DBNAME?sslmode=require
## 2) Identifie le type de backup
- Si ton fichier contient du SQL lisible (tu vois CREATE TABLE , INSERT INTO ), c’est un backup “plain SQL” → .sql
- Si c’est un fichier binaire ( .dump , .backup , .tar , .custom ) fait par pg_dump -Fc ou pgAdmin → custom format → utiliser pg_restore
## 3) Restore d’un .sql (plain SQL)
### Option A (simple) : la base existe déjà
```
$env:PGPASSWORD="TON_MDP"
psql -h HOST -p 5432 -U USER -d DBNAME -f 
"C:\chemin\backup.sql"
```
### Option B : le backup crée la base lui-même
Si ton .sql contient CREATE DATABASE ... , tu dois restaurer sur la base système postgres :

```
$env:PGPASSWORD="TON_MDP"
psql -h HOST -p 5432 -U USER -d postgres -f 
"C:\chemin\backup.sql"
```
## 4) Restore d’un .dump/.backup (custom format)
```
$env:PGPASSWORD="TON_MDP"

# (Optionnel) créer la DB vide si ton provider ne l’a 
pas déjà créée :
createdb -h HOST -p 5432 -U USER DBNAME

# Restore :
pg_restore -h HOST -p 5432 -U USER -d DBNAME -c 
--if-exists "C:\chemin\backup.dump"
```
Notes utiles :

- -c --if-exists supprime les objets existants avant de restaurer (pratique sur une base “pas propre”).
- Si tu as des erreurs de droits/ownership : ajoute souvent --no-owner --no-privileges .
Exemple :

```
pg_restore -h HOST -p 5432 -U USER -d DBNAME -c 
--if-exists --no-owner --no-privileges "C:\chemin\backup.
dump"
```
## 5) Si tu as aussi besoin des rôles/users
- Un backup pg_dump classique ne sauvegarde pas les rôles.
- Pour migrer aussi les rôles, il faut généralement un pg_dumpall --globals-only (si tu l’as).
- Sinon, recrée juste un user/password sur la nouvelle instance (souvent le provider le fait pour toi), puis restaure la DB.
## 6) Mets à jour ton projet “Révolution Network”
Dans ton code, PostgreSQL est utilisé si DATABASE_URL est défini (sinon ça bascule en SQLite). Voir database.js .

Donc après restore :

- Remplace ton DATABASE_URL par celui de la nouvelle base
- Redémarre le backend
Exemple :

```
DATABASE_URL=postgresql://USER:PASSWORD@HOST:5432/DBNAME?
sslmode=require
```
## 7) Problèmes fréquents (et solutions rapides)
- Version PostgreSQL différente : un .sql passe presque toujours, un .dump aussi mais mieux si versions proches.
- “permission denied” / owner : restaure avec --no-owner --no-privileges .
- “database does not exist” : crée la DB (ou restaure sur postgres si le .sql contient CREATE DATABASE ).
Si tu me dis juste :

- l’extension exacte de ton backup ( .sql ou .dump/.backup )
- et le provider (Render/Supabase/…)
  je te donne la commande exacte “copier-coller” adaptée à ton cas.

OK, avec un fichier .dump tu dois utiliser pg_restore (pas psql -f ).

## 1) Pré-requis
Installe PostgreSQL (au moins les outils) sur ton PC, puis vérifie que ces commandes marchent dans PowerShell :

```
pg_restore --version
psql --version
```
Si ça ne marche pas, ajoute le dossier ...\PostgreSQL\<version>\bin dans le PATH, ou lance les commandes depuis ce dossier.

## 2) Crée ta nouvelle base (vide)
Sur ton nouveau PostgreSQL, crée une DB vide (souvent via le panneau du provider).
 Sinon en ligne de commande :

```
$env:PGPASSWORD="TON_MDP"
createdb -h HOST -p 5432 -U USER DBNAME
```
## 3) Restore du .dump dans la DB
Commande “safe” (recommandée) :

```
$env:PGPASSWORD="TON_MDP"
pg_restore -h HOST -p 5432 -U USER -d DBNAME -c 
--if-exists --no-owner --no-privileges "C:\chemin\backup.
dump"
```
- -c --if-exists : supprime les tables/objets existants avant de restaurer (évite les conflits).
- --no-owner --no-privileges : évite les erreurs de droits/owner quand tu restaures sur un autre serveur.
## 4) Si ton provider impose SSL
Beaucoup de providers exigent SSL. Ajoute (au choix) :

### Option A (simple) : variable d’environnement
```
$env:PGSSLMODE="require"
pg_restore -h HOST -p 5432 -U USER -d DBNAME -c 
--if-exists --no-owner --no-privileges "C:\chemin\backup.
dump"
```
### Option B : via l’URL (si tu l’utilises)
Dans ton projet, la connexion se fait via DATABASE_URL . Tu peux mettre :

```
postgresql://USER:PASSWORD@HOST:5432/DBNAME?
sslmode=require
```
## 5) Vérifier que c’est bien restauré
```
$env:PGPASSWORD="TON_MDP"
psql -h HOST -p 5432 -U USER -d DBNAME -c "\dt"
```
## 6) Branche ton projet sur la nouvelle DB
Dans ton backend, PostgreSQL est utilisé si DATABASE_URL est défini (sinon il bascule en SQLite).
 Donc mets à jour DATABASE_URL (dans ton .env / variables du provider) avec la nouvelle URL, puis redémarre le backend.

Si c’est une base neuve (vide), cette commande est parfaite :

```
pg_restore --verbose --no-owner --no-privileges -d

```
Si tu restaures sur une base qui contient déjà des tables et tu veux écraser proprement :

```
pg_restore --verbose --clean --if-exists --no-owner 
--no-privileges -d $env:PGDATABASE "C:\chemin\backup.
dump"
```

```
psql -d $env:PGDATABASE -c "\dt"
```
